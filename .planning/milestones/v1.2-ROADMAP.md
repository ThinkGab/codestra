# Milestone v1.2: MCP File Transport

**Status:** ✅ SHIPPED 2026-04-27
**Phases:** 6–8
**Total Plans:** 3

## Overview

Milestone v1.2 adds file transfer capability to the Codestra swarm. The hub gains four HTTP routes for file storage (in-memory, UUID-keyed, swarm-namespaced). The MCP server exposes those routes as four Claude-callable tools. A skill document teaches workers when and how to use file transport. Phases 6–8 continue the integer numbering from v1.1 (which ended at Phase 5).

## Phases

### Phase 6: Hub File Routes

**Goal**: The hub stores and serves files per swarm with no path traversal exposure and enforces a 10 MB upload ceiling
**Depends on**: Nothing (pure hub-side, independent of MCP)
**Requirements**: FILE-01, FILE-02, FILE-03, FILE-04, FILE-09, FILE-10
**Plans**: 1 plan

Plans:
- [x] 06-01-PLAN.md — files Map + readRawBody helper + four HTTP file routes (PUT, GET single, GET list, DELETE)

**Details:**
- `const files = new Map()` keyed by UUID v4; filename is opaque metadata only — no filesystem path
- `readRawBody(req, maxBytes=10_485_760)` — streams raw bytes, rejects with BODY_TOO_LARGE on overflow
- PUT /files/:swarmId/:filename — stores file, returns `{id, filename, size, mimeType, uploadedAt}`
- GET /files/:swarmId/:filename — paginated `?offset=N&max_bytes=M`, returns `{content, offset, total_size, has_more}`
- GET /files/:swarmId — list metadata array (no content field)
- DELETE /files/:swarmId/:filename — removes from Map, returns `{deleted: true}`

### Phase 7: MCP Tool Wrappers

**Goal**: Claude workers can upload, download, list, and delete files in their swarm namespace through four MCP tools
**Depends on**: Phase 6
**Requirements**: FILE-05, FILE-06, FILE-07, FILE-08
**Plans**: 1 plan

Plans:
- [x] 07-01-PLAN.md — module-level registeredWorkerId + four MCP file tool wrappers

**Details:**
- `let registeredWorkerId` added to lifecycle state block; captured on successful swarm_register
- `file_upload` — accepts filename + content (≤50 KB text); proxies to PUT hub via hubFetch
- `file_download` — accepts filename + offset + max_bytes; proxies to GET hub; returns content + has_more
- `file_list` — proxies to GET /files/:swarmId; returns array of metadata
- `file_delete` — proxies to DELETE hub; returns `{deleted: true}`
- All tools guard: return isError:true if !registeredWorkerId before any fetch
- E2E verified: register → upload → download → list → delete → list (empty)

## Milestone Summary

**Key Decisions:**
- File storage in-memory (coherente con filosofia v1.x — no persistenza)
- Files keyed by UUID in hub Map; client filename is metadata only — no filesystem exposure
- file_download pagination schema included from day one (breaking change if retrofitted)
- Upload limit: 10 MB at hub layer, 50 KB text at MCP tool layer (LLM token constraint)
- readRawBody uses rejected flag + req.resume() drain (not req.destroy()) so PUT handler can write 413 before socket closes
- base64 encoding bug found post-Phase 7: file_download was returning base64 content — fixed in 657e3be

**Issues Resolved:**
- 413 delivery on oversized body (ECONNRESET instead of 413 on first implementation)
- base64 encoding in file_download hub route (657e3be fix)
- E2E race condition in Phase 7: MCP SDK async handlers + HTTP server startup required sequential JSON-RPC client

**Issues Deferred:**
- Namespace isolation enforcement (any worker can read other swarm files) — acceptable on trusted LAN
- TTL / automatic cleanup of in-memory files
- Binary file upload via two-step protocol (MVP limits to text ≤50 KB)
- File overwrite history / versioning
- UAT manual tests (10 items): require dedicated Claude Code session with Codestra MCP active

**Technical Debt Incurred:**
- None significant — clean implementation following established hub.mjs/mcp-server.mjs patterns
