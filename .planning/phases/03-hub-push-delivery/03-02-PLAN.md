The complete updated POST /messages handler:

```js
"POST /messages": async (req, res) => {
  const body = await readBody(req);
  if (!body.from || !body.to || !body.body) {
    return json(res, 400, { error: "Required fields: from, to, body" });
  }
  const msg = {
    id: generateId(),
    from: body.from,
    to: body.to,   // worker id or "broadcast"
    body: body.body,
    timestamp: new Date().toISOString(),
    readBy: new Set(),
  };
  messages.push(msg);
  // D-03: response contract unchanged — {ok, message}
  // D-07: respond FIRST, push async — sender never hangs waiting for worker
  json(res, 201, { ok: true, message: msg });

  setImmediate(async () => {
    if (msg.to === "broadcast") {
      // D-01: push to ALL workers with a callback_url; per-worker silent fallback
      const targets = [...workers.values()].filter((w) => w.callback_url);
      await Promise.allSettled(
        targets.map(async (worker) => {
          const ok = await pushToWorker(worker, msg);
          if (ok) msg.readBy.add(worker.id); // D-02: mark delivered per worker
        })
      );
    } else {
      // Unicast: push to the specific recipient if they have a callback_url
      const worker = workers.get(msg.to);
      if (worker) {
        const ok = await pushToWorker(worker, msg);
        if (ok) msg.readBy.add(worker.id); // D-02: mark delivered
        // D-05: if !ok, message stays in store-and-forward (readBy unchanged)
      }
    }
  });
},
```

Key points:
- `json(res, 201, ...)` is called BEFORE `setImmediate` — response reaches sender immediately (D-07).
- `setImmediate` defers push to the next event loop tick; does not block the HTTP response.
- Broadcast uses `Promise.allSettled` so one worker failure does not cancel others (D-01).
- Push success: `msg.readBy.add(worker.id)` — worker will not receive duplicate via polling (D-02, D-06).
- Push failure: `pushToWorker` returns false, readBy is not updated, message stays in store-and-forward (D-05).
- Workers without `callback_url` are silently skipped in both unicast (worker record found but callback_url null → pushToWorker returns false immediately) and broadcast (`.filter(w => w.callback_url)` excludes them).
- No changes to mcp-server.mjs — worker HTTP handler is already correct from Phase 2.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| hub→worker (setImmediate push) | Hub makes outbound POST to callback_url stored from registration; occurs asynchronously after 201 is sent |
| broadcast fan-out | Hub iterates all registered workers; a malicious worker registration with a crafted callback_url could redirect pushes |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-05 | Denial of Service | Promise.allSettled broadcast | mitigate | Each pushToWorker has AbortSignal.timeout(5000); allSettled never blocks longer than 5s per worker |
| T-03-06 | Spoofing | Broadcast recipient identity | accept | LAN-only scope; all workers share SWARM_SECRET; no additional identity verification needed at this milestone |
| T-03-07 | Information Disclosure | msg payload to broadcast workers | accept | Broadcast is an explicit sender choice; all LAN workers are trusted peers |
| T-03-08 | Tampering | readBy.add called on push success | accept | readBy is in-process memory; no external mutation vector within LAN trust boundary |
</threat_model>
