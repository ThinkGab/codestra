  <files>servers/hub.mjs</files>

  <read_first>
    - servers/hub.mjs — leggere l'intero file per capire il pattern dei route handler e della funzione helper `json(res, status, data)`, in particolare le righe 160–175 per il contesto della route DELETE esistente
  </read_first>

  <behavior>
    - Test 1: `workers.delete("unknown-id")` restituisce `false` → la route chiama `json(res, 404, { error: "Worker not found" })` e NON prosegue
    - Test 2: `workers.delete("existing-id")` restituisce `true` → la route chiama `json(res, 200, { ok: true })` senza il campo `deleted`
  </behavior>

  <action>
Aprire `servers/hub.mjs`. Trovare il handler della route DELETE (circa riga 170):

```js
"DELETE /workers/:id": (_req, res, params) => {
  const deleted = workers.delete(params.id);
  json(res, 200, { ok: true, deleted });
},
```

Sostituirlo con:

```js
"DELETE /workers/:id": (_req, res, params) => {
  const deleted = workers.delete(params.id);
  if (!deleted) return json(res, 404, { error: "Worker not found" });
  json(res, 200, { ok: true });
},
```

Dettagli critici:
- Il branch `if (!deleted)` usa `return` per interrompere l'esecuzione
- Il body del 404 è esattamente `{ error: "Worker not found" }` (corrispondente al pattern degli altri 404 nel file, es. riga 161)
- Il body del 200 è `{ ok: true }` senza il campo `deleted` — il campo era ridondante (sempre `true` nel path di successo)
- NON modificare nient'altro nel file
  </action>
