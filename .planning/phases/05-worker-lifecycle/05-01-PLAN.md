<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| swarm_register tool input → hub POST body | `swarmId` param flows from Claude tool call into hub registration payload as the `id` field |
| hub HTTP response → polling loop | poll responses from hub are written directly to stdout without sanitization |
| stdin pipe close → cleanup | shutdown signal originates from Claude Code process lifecycle (trusted) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-01 | Spoofing | swarm_register `swarmId` param | MEDIUM | accept | `swarmId` is an optional convenience override for the env var; the hub assigns the final ID on registration. A bad actor with MCP tool access can already do far worse — this is not an escalation. |
| T-05-02 | Denial of Service | `setInterval` polling every 10s | LOW | accept | Polling only fires when `callbackUrl` is absent (never in normal operation). 10s interval is non-aggressive; hub /messages endpoint is a cheap in-memory read. Rate limiting belongs at hub layer (out of scope v1.1). |
| T-05-03 | Information Disclosure | `[worker-poll]` stdout output | LOW | accept | Poll output goes to stdio transport (Claude Code session only). No PII expected in swarm messages. Same risk profile as existing `[worker-push]` output. |
| T-05-04 | Denial of Service | stdin 'close' not firing on all platforms | LOW | mitigate | `process.stdin.on('close')` is the canonical Node.js mechanism and fires reliably on Linux/macOS when the parent process closes the pipe. Windows behavior is consistent for stdio-based processes. Acceptable risk: worst case is a surviving process, not a security breach. No additional mitigation needed beyond the chosen mechanism (D-08). |
| T-05-05 | Elevation of Privilege | Unbounded `swarmId` string length | LOW | mitigate | Hub POST body includes raw `swarmId` as `id` field. Hub must validate/truncate ID length on ingestion (existing hub responsibility — not in scope of this plan). Executor note: do not add server-side validation here; flag for hub layer. |
</threat_model>
